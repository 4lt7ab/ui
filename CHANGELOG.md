- Add `<FormLayout>` organism to `@4lt7ab/ui` — the sectioned-form shell with a sticky save/cancel action bar and an opt-in dirty-state gate that every settings-style page reassembles by hand today. Compound API per the v0.7.0 behavior-heavy organism design doc (`01KPHD5SXPHK7GENT5SKSNW08T`): `FormLayout.Root / .Header / .Section / .SectionHeader / .SectionBody / .Actions / .SaveButton / .CancelButton / .DirtyOnChange / .NavigationGuard`, plus a public `useFormLayout()` hook. Deliberately NOT a form library — Root does not own field values, validation, serialization, or submission; consumers bring React Hook Form, Formik, TanStack Form, or plain `useState`, and FormLayout is the chrome around the form plus the dirty-state gate. Per the resolved design-doc open question, dirty-state detection is **consumer-reported, not sniffed**: Root takes the standard controlled/uncontrolled parity on both `dirty` and `saving` (pair with `onDirtyChange` / `onSavingChange` to own the state; omit for uncontrolled with `defaultDirty` / `defaultSaving`). Controlled consumers wire `dirty={formState.isDirty}` from their form library in one line; the vanilla-React escape hatch is `<FormLayout.DirtyOnChange>` — a `display:contents` wrapper that listens for bubbled `change` events and calls `setDirty(true)` on the first change, then short-circuits so subsequent keystrokes don't re-render the form. Root renders a native `<form noValidate>` (consumers opt back into HTML constraint validation by setting `noValidate={false}`) with `data-state="dirty" | "pristine"`; on submit it prevents default, invokes `onSave`, and auto-drives the uncontrolled `saving` flag while an `onSave` Promise is in flight. `FormLayout.SaveButton` is `type='submit'` with the form id wired via the `form` attribute (so it participates in submission even when viewport-sticky portals it outside the `<form>` subtree), auto-disabled when `!dirty` unless the consumer explicitly overrides via the `disabled` prop, and displays `savingLabel` + `aria-busy='true'` while `saving`; `CancelButton` fires Root's `onCancel` and chains the consumer's own `onClick` first so `preventDefault` can block the cancel. Sticky positioning has three modes via Root's `sticky` prop: `'container'` (default) uses `position: sticky; bottom: 0` to stick to the Root's scroll container — works inside a modal or DetailPage body; `'viewport'` portals Actions into a fixed-position bar at the bottom of `document.body` to escape `overflow: hidden` ancestors; `false` renders Actions inline at its JSX position for small-card cases where sticky is awkward. Actions renders `role='toolbar' aria-label='Form actions'` with `data-state='saving' | 'idle'`. `FormLayout.NavigationGuard` attaches a `beforeunload` listener while `dirty` is true and sets `event.returnValue`; modern browsers ignore the custom string but still show their canned unsaved-changes prompt (the `message` prop is preserved for legacy-browser support). Router-level navigation blocking is **not** shipped — it's router-specific and the library cannot own it without pulling a router dependency; the README documents the React Router `useBlocker` recipe. The public `useFormLayout()` hook is the first hook exposed from an organism compound: it projects the context to `{ dirty, setDirty, saving, setSaving }` so consumers can render custom UI (an "Unsaved changes" badge outside the Actions bar, a `*` next to a dirty field, a consumer-driven submit spinner). Section wires `<section aria-labelledby={sectionHeaderId}>` automatically via a nested context — SectionHeader registers its generated id on mount. Header forwards to `<Header level='page'>` for page-level forms. 23 tests cover orphan-check on every sub-part (Header / Section / SectionHeader / SectionBody / Actions / SaveButton / CancelButton / DirtyOnChange / NavigationGuard plus `useFormLayout()`), dirty-state propagation in both controlled and uncontrolled modes (including DirtyOnChange's short-circuit-once-dirty behavior), SaveButton form-id wiring + type=submit + explicit-disabled override + onSave invocation, CancelButton onCancel wiring, `useFormLayout()` hook values, NavigationGuard beforeunload listener attach/detach tied to `dirty` and preventDefault firing on real events, Actions toolbar landmark + `data-state` transitions, and the viewport-sticky portal escaping the form subtree. Registered in the demo explorer under Form Controls with the vanilla-React + controlled-dirty + sticky=false recipes (task `01KPHBRJARB9936CF9FDXWCEGN`)
